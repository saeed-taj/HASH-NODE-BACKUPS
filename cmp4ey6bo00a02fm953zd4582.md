## The Problem Nobody Talks About

Every ML tutorial starts the same way: *"First, load your dataset."*

But what do you do when the dataset doesn't exist?

That was my exact situation when I started building **CivicAI** — an AI-powered municipal complaint management system that automatically classifies citizen complaints, scores their priority, and maps them geospatially. I searched everywhere for a dataset of municipal complaints with priority labels, severity scores, and area importance. Nothing came close to what I needed.

So I did what any engineer should do , I built the dataset myself.

This is Part 1 of the CivicAI series. No models yet, no APIs, no frontend. Just the unglamorous, deeply important work of getting your data right. Because if your data is broken, everything built on top of it is broken too.

* * *

## What I Needed vs. What I Had

I started with a raw CSV of around **200 civic complaint records**. It had the basics:

*   `issue_description` — plain text describing the complaint
    
*   `location_street_address` — where the issue was reported
    
*   `resident_name` — who reported it
    
*   `is_anonymous` — whether the reporter wanted to stay anonymous
    

But it was **missing everything ML needs to learn from**:

| Column Needed | Why It's Needed | Status |
| --- | --- | --- |
| `priority_level` | Target label for Random Forest classifier | ❌ Missing |
| `priority_score` | Numeric score for model training | ❌ Missing |
| `severity_score` | Feature: how dangerous is the issue | ❌ Missing |
| `area_importance` | Feature: how critical is the location | ❌ Missing |
| `citizen_reports_count` | Feature: how many people reported this | ❌ Missing |
| `resident_email` | Required for complete resident profile | ❌ Missing |
| `report_id` | Unique identifier per complaint | ⚠️ Malformed |
| `resident_id` | Unique identifier per resident | ⚠️ Malformed |

This wasn't just missing data , it was a structural gap. The columns that would become my ML features and labels simply didn't exist yet.

## Step 1 : Filling Missing Emails

The first task was simple but important:

residents had names but no emails. The logic was straightforward, if the resident chose to report anonymously, leave the email empty. If they gave their name, derive a realistic email from it.

```python
import pandas as pd
import numpy as np
import random

df = pd.read_csv('civic_complaints_100_v4.csv')

def create_email(name, is_anonymous):
    if is_anonymous or pd.isna(name):
        return np.nan
    return f"{name.lower().replace(' ', '.')}@{random.choice(['gmail.com', 'outlook.com', 'yahoo.com'])}"

df['resident_email'] = df.apply(
    lambda row: create_email(row['resident_name'], row['is_anonymous']),
    axis=1
)

df.to_csv('civic_complaints_100_v4.csv', index=False)
```

**Why this matters:** In a real municipal system, email is how the city communicates back to residents , update on their complaint, resolution notice, etc. Having no email for non-anonymous reporters was a data quality issue that would affect the product, not just the model.

## Step 2 : The Core Challenge: Engineering Ground Truth Labels

This is where things got interesting. For supervised ML, you need labels. My dataset had none of the priority related columns. I had to **engineer them from scratch using domain logic** , these became my *ground truth*.

In industry, ground truth is often established by domain experts. In my case, I thought through what a municipal officer would actually consider when prioritising a complaint and translated that into formulas.

### Feature 1 : Severity Score (1–10)

Severity measures how dangerous or urgent the issue itself is, based purely on the language used in the complaint description.

```python
desc = str(row['issue_description']).lower()

if any(word in desc for word in ['accident', 'dangerous', 'injured', 'flooding', 'fire', 'collapse']):
    severity = random.randint(8, 10)

elif any(word in desc for word in ['overflowing', 'broken', 'large', 'not working']):
    severity = random.randint(5, 7)

else:
    severity = random.randint(1, 4)
```

**The reasoning:**

*   Words like *"fire"*, *"collapse"*, *"injured"* indicate life-threatening situations : severity 8–10
    
*   Words like *"broken"*, *"overflowing"* indicate functional failures : severity 5–7
    
*   Everything else , cosmetic issues, minor inconveniences : severity 1–4
    

This is NLP at its simplest form: keyword-based intent detection. No model yet, just domain logic encoded in code.

### Feature 2 : Area Importance (1–10)

A pothole outside a hospital is far more dangerous than the same pothole in an empty park. Area importance captures this , the same complaint gets a different weight depending on *where* it is.

```python
loc = str(row['location_street_address']).lower()

if 'hospital' in loc:                        area_imp = 10
elif 'school' in loc:                        area_imp = 9
elif 'highway' in loc or 'main' in loc:      area_imp = 8
elif 'commercial' in loc or 'st' in loc:     area_imp = 7
elif 'park' in loc:                          area_imp = 5
else:                                        area_imp = random.randint(2, 3)
```

**The reasoning:**

| Location Type | Score | Why |
| --- | --- | --- |
| Hospital | 10 | Any obstruction near a hospital is life-critical |
| School | 9 | Children are vulnerable, high foot traffic |
| Highway / Main road | 8 | High vehicle volume, accident risk |
| Commercial / Street | 7 | Economic impact, public visibility |
| Park | 5 | Lower urgency, no critical infrastructure |
| Other | 2–3 | Residential or unclassified areas |

### Feature 3 : Citizen Reports Count

How many people reported the same issue? A pothole reported by 40 citizens is more urgent than one reported by 1.

I simulated this with a random integer between 1 and 50, representing real-world reporting frequency.

```python
count = random.randint(1, 50)
```

In a production system, this would come from a deduplication layer that groups similar complaints. For training purposes, simulated variance is enough.

## Step 3 : The Priority Score Formula

With all three features calculated, I combined them into a single weighted formula:

```plaintext
priority_score = (0.4 × severity) + (0.3 × normalized_citizen_count) + (0.3 × area_importance)
```

The citizen count gets normalized first so it's on the same scale (0–10) as the other features:

```python
norm_count = (count / 50) * 10
priority_score = (0.4 * severity) + (0.3 * norm_count) + (0.3 * area_imp)
```

**Weight distribution rationale:**

| Feature | Weight | Reasoning |
| --- | --- | --- |
| Severity | 40% | The nature of the issue is the strongest signal |
| Citizen Count | 30% | Community impact matters — if many report it, act faster |
| Area Importance | 30% | Location context is equally important as community signal |

### Mapping Score to Priority Level

```python
if priority_score < 4.0:   level = 'low'
elif priority_score < 7.0: level = 'medium'
elif priority_score < 8.5: level = 'high'
else:                       level = 'critical'
```

These thresholds weren't arbitrary , they were designed so the distribution across levels would be roughly balanced, avoiding a dataset where 90% of rows are "low" (which would make a classifier useless).

The full function applied to every row:

```python
def infer_and_calculate_priority(row):
    desc = str(row['issue_description']).lower()
    loc = str(row['location_street_address']).lower()
    
    # Severity
    if any(word in desc for word in ['accident', 'dangerous', 'injured', 'flooding', 'fire', 'collapse']):
        severity = random.randint(8, 10)
    elif any(word in desc for word in ['overflowing', 'broken', 'large', 'not working']):
        severity = random.randint(5, 7)
    else:
        severity = random.randint(1, 4)
    
    # Area Importance
    if 'hospital' in loc:                        area_imp = 10
    elif 'school' in loc:                        area_imp = 9
    elif 'highway' in loc or 'main' in loc:      area_imp = 8
    elif 'commercial' in loc or 'st' in loc:     area_imp = 7
    elif 'park' in loc:                          area_imp = 5
    else:                                        area_imp = random.randint(2, 3)
    
    # Citizen Count + Score
    count = random.randint(1, 50)
    norm_count = (count / 50) * 10
    priority_score = (0.4 * severity) + (0.3 * norm_count) + (0.3 * area_imp)
    
    # Priority Label
    if priority_score < 4.0:   level = 'low'
    elif priority_score < 7.0: level = 'medium'
    elif priority_score < 8.5: level = 'high'
    else:                       level = 'critical'
    
    return pd.Series([level, round(priority_score, 2), severity, area_imp, count])

cols = ['priority_level', 'priority_score', 'severity_score', 'area_importance', 'citizen_reports_count']
new_data = original_df.apply(infer_and_calculate_priority, axis=1)
new_data.columns = cols

final_df = pd.concat([original_df, new_data], axis=1)
final_df.to_csv('civic-issue-reporting-dataset.csv', index=False)
```

## Step 4 : Fixing Malformed IDs

The original dataset had `report_id` and `resident_id` columns but they were inconsistently formatted, some rows had them, some didn't, formats varied. I dropped and regenerated both cleanly.

```python
df = df.drop(columns=['report_id', 'resident_id'], errors='ignore')

# Unique report ID per complaint
report_ids = random.sample(range(100000, 999999), len(df))
df['report_id'] = [f"RPT-{i}" for i in report_ids]

# Resident ID — same person always gets the same ID
unique_residents = df['resident_name'].unique()
random_ids = random.sample(range(1000, 9999), len(unique_residents))
mapping = {name: f"RES-{rid}" for name, rid in zip(unique_residents, random_ids)}
df['resident_id'] = df['resident_name'].map(mapping)
```

**Key insight here:** `report_id` is per row (always unique), but `resident_id` is per *person* (same resident = same ID across multiple complaints). Getting this wrong would break any future analysis of repeat reporters.

## Step 5 : Growing the Dataset from 200 to 1100 Rows

200 rows is not enough to train a robust classifier. A Random Forest on 200 samples would memorise the training data rather than learn patterns , classic overfitting.

I needed more data, specifically more **high** and **critical** priority examples, since those are naturally rarer in real complaint data. I used **Gemini** to generate 4 batches of synthetic complaints:

*   `Final_Batch_1_High.csv` — High priority complaints
    
*   `Final_Batch_2_High.csv` — High priority complaints
    
*   `Final_Batch_3_Critical.csv` — Critical priority complaints
    
*   `Final_Batch_4_Critical.csv` — Critical priority complaints
    

Then merged everything and shuffled:

```python
df_original = pd.read_csv('municipal_complaints_v2.csv')

b1 = pd.read_csv('Final_Batch_1_High.csv')
b2 = pd.read_csv('Final_Batch_2_High.csv')
b3 = pd.read_csv('Final_Batch_3_Critical.csv')
b4 = pd.read_csv('Final_Batch_4_Critical.csv')

df_combined = pd.concat([df_original, b1, b2, b3, b4], ignore_index=True)

# Shuffle — critical to prevent the model from learning row order
df_shuffled = df_combined.sample(frac=1, random_state=42).reset_index(drop=True)

df_shuffled.to_csv('municipal_training_set_1100.csv', index=False)

print(f"Total rows: {len(df_shuffled)}")
```

**Why shuffle?** If your training data has all low-priority rows first and all critical rows at the end, the model learns order rather than features. `random_state=42` makes the shuffle reproducible, anyone running this code gets the same result.

## What I Learned

**1\. No dataset is a feature, not a blocker.**

Building my own data forced me to deeply understand the domain — what actually makes a complaint high priority, what location features matter, how severity should be measured. That understanding directly improved my model design.

**2\. Ground truth is a design decision.**

The formulas I wrote — the weights, the thresholds, the keyword lists are all choices I made. Different choices would produce a different model. Being explicit about those choices is what makes a system auditable and trustworthy.

**3\. Class balance is a preprocessing problem, not a model problem.**

I added synthetic high and critical rows before training, not after. Trying to fix imbalance at the model level (with class weights or resampling) is a workaround. Fixing it in the data is cleaner.

**4\. ID design matters.**

The distinction between a per-row ID (`report_id`) and a per-entity ID (`resident_id`) is a database design concept that appears in data preprocessing too. Getting it wrong corrupts every analysis that depends on it.
